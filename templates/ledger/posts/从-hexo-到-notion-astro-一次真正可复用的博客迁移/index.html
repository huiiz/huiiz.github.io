从 Hexo 到 Notion + Astro：一次真正可复用的博客迁移

这次我把博客从 Hexo 迁移到 Notion + Astro，不只是换技术栈，而是把内容管理、资源组织、自动部署和可复用的部署流程一起重构了一遍。

发布于 2026年4月26日 阅读约 10 分钟
技术分享
#Hexo#Notion#博客#建站

为什么我要把博客从 Hexo 迁出来

我之前的博客一直跑在 Hexo 上。作为静态博客工具，它其实很好用：主题成熟、社区足够大、GitHub Pages 部署也很顺手。问题不是它不能用，而是它越来越不符合我现在的写作和维护方式。

随着文章变多，我开始越来越频繁地在几个东西之间来回切换：本地 Markdown、图片目录、主题配置、部署命令、GitHub Actions。只要是一次完整写作，这套流程还能接受；但如果我只是想改一段摘要、补一张图、发一个草稿，它就显得有点重。

所以这次迁移，本质上不是为了追新，而是为了让博客真正回到“内容优先”。我希望我维护的是文章，而不是一堆围着文章转的本地文件和脚本细节。

为什么最后选了 Notion + Astro

我最后选的是 Notion + Astro。Notion 负责内容管理，Astro 负责静态生成，GitHub Pages 继续负责托管。这个组合对我来说很自然，因为我本来就在 Notion 里做大量记录，把博客也收进同一个内容系统之后，整个写作入口就统一了。

Notion 的优势在于编辑体验足够轻，页面结构、数据库字段、封面、标签和页面组织都很直观。Astro 的优势则在于内容驱动站点的生成效率高，结构清晰，而且不会把我拖进一套过重的前端工程复杂度里。

这两个工具组合起来之后，我就得到了一个很接近理想状态的博客系统：内容在 Notion 里维护，页面由 Astro 在构建时生成，最终仍然发布成纯静态页面。

新架构现在长什么样

现在这套博客的链路已经比较清晰了：

  • 我在 Notion 中写文章和维护页面。
  • 构建前脚本从 Notion 同步文章数据和页面内容。
  • Astro 根据这些数据生成静态页面。
  • GitHub Actions 构建并把 dist 发布到 huiiz.github.io。
  • 正式域名继续使用 blog.huii.top。

如果只看发布后的结果，它和以前一样仍然是一个静态博客。但如果看维护体验，它已经从“本地 Markdown 驱动”变成了“内容管理驱动”。这就是这次迁移真正值得的地方。

项目地址

这次迁移涉及到两个仓库，一个放源码，一个放最终发布结果：

  • 源码仓库：https://github.com/huiiz/notion-astro-journal
  • 发布仓库：https://github.com/huiiz/huiiz.github.io
  • 线上地址：https://blog.huii.top

源码仓库里保留了 Astro 项目、同步脚本、Cloudflare Worker、GitHub Actions 工作流以及完整说明文档。也就是说，这不是一套“只能我自己用”的临时方案，而是一个别人也能照着复用的项目。

自动更新也一起接好了

我不想每次在 Notion 改完内容之后，还手动跑一遍构建再推送。所以这次顺手把自动更新链路也搭好了。

  • Notion Webhook 负责在内容更新后发出通知。
  • Cloudflare Worker 负责接收通知并转发成 GitHub repository_dispatch。
  • GitHub Actions 负责重新构建并自动发布。
  • 另外还加了一个定时兜底构建，避免偶发 webhook 丢失。

这样之后，博客真正进入了一个比较舒服的状态：我只管在 Notion 改内容，站点会自己重新生成并上线。

如果你也想照着做，完整部署过程可以这样走

下面我把这次实际使用的路径按步骤写下来，尽量做到别人拿去就能照着做。

第一步：创建 Astro 项目

npm create astro@latest
cd notion-astro-blog
npm install

先有一个 Astro 项目骨架，后面的同步脚本、页面模板和部署配置都基于这个项目继续搭。

第二步：安装依赖

npm install @notionhq/client astro @astrojs/rss @astrojs/sitemap

其中 `@notionhq/client` 用于拉取 Notion 数据，`@astrojs/rss` 和 `@astrojs/sitemap` 则是博客站点常见的增强项。

第三步：准备环境变量

NOTION_TOKEN=你的_notion_token
NOTION_DATABASE_ID=你的_notion_database_id
SITE_URL=https://blog.huii.top
BASE_PATH=

如果还要部署 Cloudflare Worker 自动更新，也要额外准备 Cloudflare 的凭证。

CLOUDFLARE_API_TOKEN=你的_cloudflare_api_token
CLOUDFLARE_ACCOUNT_ID=你的_cloudflare_account_id

第四步：写构建前同步脚本

核心思路是在 Astro 构建前执行一个同步脚本，从 Notion 拉文章和页面，然后生成本地数据文件。

{
  "scripts": {
    "dev": "astro dev",
    "prebuild": "node scripts/fetch-notion.mjs",
    "build": "astro build",
    "sync:notion": "node scripts/fetch-notion.mjs"
  }
}

这样每次 `npm run build` 之前，都会先同步一次 Notion 数据。

第五步：本地同步并预览

npm run sync:notion
npm run dev

这一步很重要，因为你可以先确认文章页、页面页、封面、标签、摘要和图片路径是不是都对，再进入部署。

第六步：配置 GitHub Actions 自动发布

我的做法是把源码仓库和最终发布仓库分开。源码仓库负责构建，发布仓库负责承接 `dist/`。

name: Build and Publish Blog

on:
  push:
    branches: [main]
  repository_dispatch:
    types: [notion-content-updated]
  workflow_dispatch:

jobs:
  build-and-publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - uses: peaceiris/actions-gh-pages@v4
        with:
          personal_token: ${{ secrets.PUBLISH_TO_PAGES_TOKEN }}
          external_repository: huiiz/huiiz.github.io
          publish_branch: master
          publish_dir: ./dist
          cname: blog.huii.top

你需要在源码仓库里提前配置好 `NOTION_TOKEN`、`NOTION_DATABASE_ID` 和 `PUBLISH_TO_PAGES_TOKEN`。

第七步：接上 Notion Webhook 自动更新

如果你希望在 Notion 里改完内容后自动上线，还需要再接一层 Cloudflare Worker。

cd automation/notion-webhook-relay
npm install
npx wrangler deploy

Worker 收到 Notion Webhook 后，会转发成 GitHub 的 `repository_dispatch`，然后由 GitHub Actions 重建站点。

npx wrangler secret put GITHUB_DISPATCH_TOKEN
npx wrangler secret put NOTION_WEBHOOK_VERIFICATION_TOKEN

Notion 那边需要把 webhook 地址填成你的 Worker 地址，并完成一次 verification token 验证。

第八步：整理资源和旧内容

这一步最容易被忽略，但其实非常重要。迁移成功不代表体验就自动变好，图片、附件、封面来源、摘要长度、卡片比例这些都需要重新整理。

迁移过程中踩过的坑

这次迁移并不是完全顺滑的，几个坑还是挺典型的：

  • Notion 页面封面和数据库封面字段如果同时存在，很容易让来源混乱，最后我统一改成只使用页面封面。
  • 并不是所有附件类型都适合直接进 Notion，某些二进制资源仍然更适合保留为站点静态文件。
  • 自动部署链路虽然优雅，但第一次接 Notion Webhook、Cloudflare Worker 和 GitHub 签名校验时，排障成本并不低。
  • 旧内容即使迁移成功，也不代表阅读体验就自动变好了，很多摘要、排版和列表展示仍然需要重新整理。

最后的感受

如果说以前的 Hexo 博客更像一个“本地维护、手动整理、最后部署”的静态项目，现在这套 Notion + Astro 更像一个“内容优先、构建自动化、展示可控”的发布系统。

它不一定是最轻量的起步方案，但对我来说，它是一个更舒服的长期方案。至少到现在，我最想解决的问题已经解决了：写作更顺，发布更自动，维护也更清晰。

如果你也在考虑从 Hexo 迁出来，这条路是值得试的。不是因为它更新，而是因为它更贴近“博客的重点应该是内容”这件事。

如果你想直接参考这套方案，可以优先看源码仓库、README 和自动部署文档:https://github.com/huiiz/notion-astro-blog，再决定是否一步到位接上 Notion Webhook。
